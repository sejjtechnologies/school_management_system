How I Developed the System

Problem & goals: I built this system to reduce administrative overhead for organisations by centralising user and record management, configurable workflows, and basic financial tracking. The main goals were reliability, clarity for non-technical users, and modularity so features can be added safely across different domains.

Architecture: The backend is implemented with Flask using Blueprints to keep features separated (admin, bursar, teacher, secretary, parent). Data is managed with SQLAlchemy models; templates are rendered with Jinja2 and the UI uses Bootstrap 5 for responsive components.

Engineering choices: I kept business logic modular, favoured readability, and used small helper scripts (e.g. `hash_password.py`) for security-sensitive operations. Forms validate on the server and are progressively enhanced client-side. Payments, receipts and audit-friendly references are stored to ensure traceability across financial and operational workflows.

Quality & deployment: Passwords are hashed, role checks guard sensitive routes, and the app is ready for deployment to common Python hosts (Gunicorn/NGINX or PaaS). I recommend adding Alembic for migrations and standard backups for production use.

How It Works

The system follows simple role-based workflows so every user has clear responsibilities and permissions.

  • Admin: creates role accounts (teachers, bursars, parents, headteacher, secretary) and manages permissions. Admins can help reset accounts or update user details.
  • Secretary: logs in with the created credentials to register students, assign them to class & stream, and manage pupil records (edit/delete when needed). If a secretary forgets their password they can use the reset link or contact the admin for help.
  • Teacher: after being assigned a class, logs in to enter marks, submit reports, track attendance and print or download detailed pupil reports.
  • Bursar: records payments for pupils, generates receipts, tracks salaries and school expenses, and produces simple financial summaries.
  • Parent: logs in to view their child's progress, download receipts and communicate with school staff where enabled.

These workflows are connected: the admin provisions accounts and assignments, the secretary registers pupils and links them to classes/streams, teachers and bursars operate on the assigned pupils, and parents access reports and receipts.
